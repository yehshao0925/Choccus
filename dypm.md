# dypm 演算法技術剖析（Pommerman 2018 冠軍 agent）

> 用途：給 Choccus AI bot 對照參考。本文整理 dypm 的真實做法，並逐項標出
> 「論文/比賽有明確記載」vs「依 Pommerman 通行做法的合理重建」vs「不確定」。

---

## 0. 來源與歸屬（先正名）

- **作者／單位**：**IBM Research – Tokyo**，Takayuki Osogami（小篠 隆之）與 Toshihiro Takahashi。
  - ⚠️ 常見誤傳：dypm **不是** Borealis AI / RBC 的作品。Borealis 在 Pommerman 另有 agent。
- **成績**：奪下 **NeurIPS 2018 Pommerman**「**team（2v2）**」賽道冠軍。參賽帳號為 `dypm.1` / `dypm.2`。
- **核心論文**：Osogami & Takahashi, **〈Real-time tree search with pessimistic scenarios〉**
  （IBM Research – Tokyo, 2019；arXiv 約 1902.xxxxx，發表於 ACML 2019 一帶）。
  - ⚠️ 確切 arXiv 編號與會議我不完全確定；引用前請查證。論文的**核心機制描述**則是可信的。
- 名稱 "dypm" 的字面含義我不確定，不臆造。

**一句話定位**：dypm = **「以遊戲自身 forward model 做即時、深度受限的前向搜尋，並用『悲觀情境（pessimistic scenarios）』取代期望值，來保證最壞情況下不被炸死」的純搜尋型 agent（無 RL）**。

---

## 1. 整體決策架構

dypm **不是**單步加權評分迴圈，而是**深度受限的前向搜尋（depth-limited forward-model search）**，但它的搜尋形狀和一般 minimax/MCTS 很不一樣：

- **只展開「自己」的動作樹**：節點分支是 agent 自己的 6 個動作（停、上、下、左、右、放彈）。
  它**不**在每個 ply 把 4 名 agent ×6 動作全展開（那是 minimax 的指數爆炸來源）。
- **對手與環境的未來，用「情境（scenario）」處理**，而不是用樹分支處理。每個情境是一組
  「對手會怎麼動、炸彈會何時爆」的**確定性假設**。同一棵自走樹會在數個情境下各被評估一次。
- **即時限制**：Pommerman 每步 **100ms** 硬上限 → 搜尋深度受限（通常數步到十幾步的數量級，
  視局面與剪枝而定；⚠️ 確切深度論文未給死數字，是 anytime / 受時間預算控制）。

> 對照你目前的 bot：你的「dypm 式單一加權評分迴圈」其實是把 dypm 的**葉節點評估函數**
> 抽出來當成單步決策；dypm 真正的骨幹多了**前向多步模擬 + 悲觀情境的最壞情況保證**這層。
> 這正是可以拿來強化 Choccus bot 的方向（見 §10）。

---

## 2. Forward model search

- **模擬器 = 遊戲規則本身**。Pommerman 提供確定性 forward model（給 state + 所有人的 action →
  下一個 state）。dypm 直接用它逐步推進，**不需要學習一個近似模型**。這是它能純搜尋的前提。
- **分支**：每個節點只列舉「我方」動作。對手動作不在這裡分支，而是由「當前情境」指定（§3）。
- **深度／預算**：anytime，吃滿 100ms。實務上靠 §4 的危險圖把「明顯送死」的分支早早砍掉，
  把預算集中在還活著的路線上。
- **葉節點**：到達深度上限或局勢明朗時，用啟發式評估函數打分（§6）。

## 3. 悲觀情境（Pessimistic Scenarios）— 全篇核心

### 3.1 動機
Pommerman 最大的死因是**自殺 / 被連鎖波及 / 被對手封逃生路**。若用 **expectimax**（對對手動作取
**期望/平均**），agent 會「平均看起來安全」就行動，但只要**有一條**最壞分支會死，它就死了——
平均值掩蓋了致命尾端風險。

dypm 的答案：**別取平均，取最壞（min）**。

### 3.2 什麼是「情境」
一個 scenario = 把所有不確定性**釘死成一組確定假設**，例如：
- 對手會往對我最不利的方向移動 / 會在能封我退路的位置放彈；
- 看不到引信的炸彈，**以最壞的爆炸時機**計入；
- 連鎖爆炸**取最早可能引爆 tick**。

把多個這類假設組成一個小的、**人工定義的情境集合 S**（數量是個位數，而非全動作展開）。

### 3.3 評估方式：maximin（robust optimization）
對我方候選首步動作 a：

```
value(a) = min_{s ∈ S}  forwardSearchValue(a, s)      # 對情境取最壞
chosen   = argmax_a  value(a)                          # 在最壞情況下選最好
```

這是 **maximin / 穩健最佳化**：先假設世界跟你作對（min），再從「最壞情況最不糟」的動作裡挑（max）。
- **保命項**用悲觀（min）→ 保證所選動作在所考慮的最壞情境下**仍有活路**。
- ⚠️ 細節：論文不是對所有目標都純悲觀。**survivability 用悲觀**，**reward/進攻**可用較樂觀或加權
  估計，最後組合。確切的加權形式我不逐字確定，但「**安全悲觀、收益樂觀**」的二分是可信的精神。

### 3.4 與其他方法的根本差異

| 方法 | 對手未來怎麼處理 | 致命弱點 |
| --- | --- | --- |
| **Expectimax** | 對手動作取**機率加權平均** | 平均掩蓋致命尾端 → Pommerman 裡常自殺 |
| **Minimax** | 每 ply **全展開**對手所有動作取 min | 4 agent×6 動作指數爆炸，100ms 內搜不深 |
| **MCTS（純）** | 隨機 rollout 取平均 | 樣本少時對「罕見但致命」事件估計極差 |
| **dypm（悲觀情境）** | 只展開**自己**的樹；對手由**少數釘死的最壞情境**代表，取 **min** | 情境集若漏掉某種最壞行為，仍可能失手；情境是人工設計 |

**關鍵洞見**：悲觀情境用「**少量手工最壞情境 + 取 min**」近似了 minimax 的安全保證，
卻避開了 minimax 的分支爆炸——這才是它的本質，不是「為效率而剪枝」。

## 4. Reachability / Survivability 分析

這是 dypm（與所有強 Pommerman agent）的工程核心。

### 4.1 危險圖（flame timing map）
對每個格子算出「**何時會著火、著火持續到何時**」：
- 掃描場上每顆炸彈，依火力沿十字射線標記受影響格與其爆炸 tick。
- **連鎖**：一顆炸彈的火會引爆射程內其他炸彈 → 連鎖中每顆取**最早可能引爆 tick**。
- 射線**遇硬牆停、炸到軟牆停**（軟牆當 tick 被毀，但不再往後延伸）。
- 結果是 `dangerUntil[x][y]` / `flameAt(x,y,t)` 之類的查詢。

> 你的 Choccus CLAUDE.md 已經有對應鐵則（「danger map 的爆炸射線須逐字對齊 sim 的
> `Explosion.ts`，連鎖取最小引爆 tick、殘留 `[t, t+27)`、軟磚停格不殘留」）——
> 這正是 dypm 這層的等價物，方向完全正確。

### 4.2 Survivability（可活性）
一格-時刻對 `(x, y, t)` 是「**可活**」若：
1. 該時刻不在火上（查危險圖），且
2. 從現在位置存在一條**逐 tick 都不踩火**的移動序列到得了那裡（對自走樹做 BFS/前向展開）。

agent 維持「**至少存在一條 guaranteed-safe 路徑**」這個不變量。**在悲觀情境下**仍滿足此不變量的
動作，才被視為安全。

## 5. 放炸彈安全閘門（bomb-placing safety gate）

放彈是最容易自殺的動作，dypm 對它有**硬性前置檢查**：

> 「放下這顆炸彈之後，**在悲觀情境下仍存在一條保證逃生路徑**，到得了一個全程不致命的格子」——
> 否則此放彈動作直接淘汰，連評分都不進。

具體：
1. 假想放下炸彈 → 把它加進危險圖（含它會引發的連鎖）。
2. 在**悲觀情境**（對手可能來封路、其他炸彈以最壞時機爆）下，做 survivability BFS。
3. 找不到保證活路 → **禁放**。

這個閘門是「不自殺」的第一道、也是最強的一道防線（評分函數是第二道）。

> 你 CLAUDE.md 的「放彈前過硬性安全閘門（必須存在逃生路徑到視野內全程不致命的格，否則該動作淘汰）」
> 與此**一模一樣的精神**。差別只在 dypm 的閘門跑在「悲觀情境」上（考慮對手主動封路），
> 你的版本若只考慮現有炸彈、不考慮對手會來堵，安全性會略弱（見 §10）。

## 6. 評估／效用函數（葉節點打分）

通過安全閘門後，dypm 用啟發式評估在**安全動作集**裡選最佳。打分特徵（典型、與描述相符）：

- **存活/逃生餘裕**：逃生路徑長度、可達安全格數量（越多越好）。
- **撿道具**：到最近 powerup（火力/炸彈/速度）的距離（越近越好）。
- **炸軟牆/開路**：一次放彈能炸到的軟牆數（開地圖、找道具、製造攻擊角）。
- **對敵壓制**：與敵人的距離 / 能否把敵人逼進死路或封其退路（trap）。
- **位置價值**：避免被逼到角落、保持機動。

葉節點 value = 這些特徵的加權組合（reward 面可用較樂觀估計）。
⚠️ 確切權重與特徵清單論文未逐一公開係數；上述是「方向可信、係數不保證」。

## 7. 如何避免自殺（Pommerman 頭號死因）

dypm 的「不自殺」是**多層**的，不是單一技巧：

1. **危險圖**精準（含連鎖、最壞時機）→ 知道哪裡何時會死。
2. **survivability 不變量**：每步都要保留至少一條 guaranteed-safe 路徑。
3. **放彈安全閘門**：放彈後仍須有保證活路，否則禁放。
4. **悲觀情境（取 min 不取平均）**：把「對手主動害你、炸彈以最壞時機爆」當預設 →
   不會因為「平均安全」而踏進致命尾端。

第 4 點是 dypm 贏過 RL agent 的關鍵：很多 RL/expectimax agent 在「平均安全」的局面自信前進，
被一條低機率但致命的分支收掉；dypm 因為**結構性地拒絕踏入任何最壞會死的動作**，自殺率極低。

## 8. 對手建模

- dypm **不訓練、不學習對手的策略模型**。
- 它把對手當成**情境集合裡的最壞行為**來對待——即**對抗式 / 悲觀**處理，而非機率預測。
- 好處：**不可被 exploit**、不需訓練資料、對沒見過的對手也穩（這正是它打贏學習型 agent 的原因之一）。
- ⚠️ 是否帶任何輕量啟發式對手預測（如「對手傾向往道具走」）我不確定；
  即使有也是次要，主軸是悲觀對抗。
- **隊友（2v2）**：team 賽道需避免炸到隊友、隱性配合；dypm 把隊友納入 forward model，
  至少做到「不互炸」。⚠️ 是否有更主動的協同（夾擊）我不確定。

---

## 9. 決定性與「為何不用 RL」

- **純搜尋 + 啟發式，零 RL**。Forward model 用遊戲規則本身，過程**確定性**。
- 作者的論點（可信、與賽果一致）：Pommerman 對 RL **極不友善**——
  - **稀疏 + 誤導性獎勵**：只有輸贏訊號；
  - **credit assignment 災難**：你可能是被**隊友的炸彈**炸死，或自己 100 步前的放彈造成；
  - **對手非平穩**、**高方差**：學到的策略易過擬合、易被 exploit。
  - 結果：當年多支 RL agent 連「穩定不自殺」都做不到。
- 用**好的 forward model + 悲觀搜尋**，把「不死」變成**結構保證**而非「學出來的傾向」，
  在 100ms 即時預算下又跑得動 → 這就是 dypm 的工程取捨，也是它奪冠的核心理由。

## 10. 對 Choccus bot 的具體啟示（可選的強化方向）

你現有 bot（單步加權評分 + 危險圖 + 放彈安全閘門）**已抓到 dypm 的 60–70% 精髓**。
若要更接近 dypm，差距與可選補強：

1. **從「單步」走向「淺前向搜尋」**：對自己的動作做 3–6 步 forward search，葉節點才套你現在的
   評分函數。能避免「這步看起來好、下步必死」的短視。（成本：sim 要可快速 rollback / 複製。）
2. **把安全閘門升級成「悲觀情境」版**：目前你的逃生驗證若只考慮**現有炸彈**，建議加入
   **「對手可能在我逃生路上補一顆彈 / 來堵路」**的最壞假設，至少對「敵人 ≤N 格且我正要進窄道」時。
   這正是 dypm 比一般 agent 穩的地方。
3. **survivability 以 min（最壞）聚合、別用平均**：tie-break 與安全判定一律取最壞情境，
   契合你「完全決定性、tie-break 順序固定」的鐵則。
4. **維持你已有的優勢**：你的 determinism 鐵則（禁 `Math.random`/`Date.now`、threaded RNG、
   danger map 逐字對齊 `Explosion.ts`）與 dypm 的「確定性 forward model」哲學完全一致——
   不要為了搜尋引入非決定性。

> 注意：CLAUDE.md 已定調 bot 為「dypm **式**單一加權評分迴圈，**非**離散模式切換」。
> 上面 §10.1 的「淺前向搜尋」是**增強葉節點之前的一層**，不違背「單一核心評分」——
> 仍是同一套評分，只是從 1 步看到數步。若要採用，請當成 AI 版本升級（v2）走 `version-bench` 驗證。
